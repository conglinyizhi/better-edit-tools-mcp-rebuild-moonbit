# 否决提案记录

> 本文档记录 better-edit-tools-mcp MoonBit 移植过程中被明确否决的提案。
> 每项包含：提案标题、提案内容、否决原因。

## 虚拟文件系统

- 为测试实现内存文件系统（MemFileSystem）：在 `fs` 包中实现一个 `MemFileSystem`，用于单元测试替代真实文件系统。

**否决原因**：

- 这个项目一开始是从 Go 语言迁移过来的，Go 语言的生态下有成熟的文件虚拟 库来用。但是在 moonbit 下并没有（也可能只是我暂时没找到）
- 内存文件系统需要自行模拟 `@fs` 的权限、换行、编码、目录创建、原子重命名等行为，容易与真实文件系统产生偏差。并且在当前生态没有找到对应库的情况下，会容易增加开发量。而且需要重新设计包的一些相关。逻辑 这对于一个既作为库又作为二进制发行的工具来说，有点臃肿
- 如果某个测试只跑在虚拟文件系统上。那么在编译后跑在真实文件系统上的失败风险仍然存在，这会让测试变得相当无用。
- MoonBit 没有 `finally`/`defer`，但可以用 `try/catch` 包装 `with_temp_dir` helper 实现真实临时目录的自动清理。
- 真实临时目录能完整捕获错误链路：工具层 → `@fs` → 系统调用 → `BeError`，更符合“测试即生产”的原则。

## 不移植 JSON 降级解析能力

**提案内容**：只实现标准 JSON 解析，不移植 Go 版本中的 `normalizeLineBreaks` 和 `parseSpecRaw` 降级提取逻辑。并且移植 JSON 降级能力会增加开发量

**否决原因**：

- `\n` 双转义是 MCP 调用链路上的常见问题，不处理会导致多行内容被压缩成一行，严重破坏编辑结果。
- 降级解析是原项目的**核心能力**之一，这将是“减少 token 浪费、挽回错误调用”的核心依赖。
- 该能力与 chip 恢复机制联动：降级写入失败后保存参数，用户可通过 `be-insert-chip` 恢复内容。缺少降级解析会打断这一闭环。
- 实现成本可控：先完成 `normalize_line_breaks`，再逐步补齐常见降级场景即可。

## 将 `be_*` 函数完全改为纯逻辑库函数，移除高层入口

**提案内容**：把 `be_read`/`be_replace` 等函数设计成只操作行数组或字符串的纯函数，不再直接读写文件；由 MCP server/CLI 自行组合这些函数。

**否决原因**：

- 会导致每个调用方（MCP server、CLI、外部包）都重复“读文件 → 调纯函数 → 写文件 → 构造结果”的样板代码。
- `be_*` 作为高层入口能统一错误处理、内容警告、diff 生成、snapshot 记录等行为，避免调用方遗漏。
- 保留高层入口的同时暴露底层纯函数，可以同时满足“工具使用”和“库复用”两种需求。

---

## 不实现 session / snapshot / chip 机制

**提案内容**：只移植基础读写和编辑工具，跳过 `viewed_code_id` session 缓存、编辑快照回滚、chip 失败参数保存等机制。

**否决原因**：

- session 是 `be-read` 与 `be-replace` 行数一致性校验的基础；缺少它，`viewed_code_id` 只是静态哈希，无法在文件变化后给出警告。
- snapshot 是 `be-trx` 工具的核心，也是项目区别于普通文件编辑工具的重要特性。
- chip 是“失败→恢复”闭环的关键；`be-insert-chip` 的设计目标就是回放失败操作的内容或参数。
- 这三者共同构成原项目的“安全网”，跳过会大幅降低产品价值。

---

## 每个工具函数使用独立可选参数，不引入统一 `Config`

**提案内容**：`be_replace` 等函数继续通过 `fs?`、`lang?`、`preview?` 等独立可选参数扩展，不使用统一的 `Config` 结构。

**否决原因**：

- 随着功能增加，函数签名会越来越长，公共 API 频繁变动。
- 外部调用方需要记忆每个函数的参数顺序和默认值。
- `Config` 可以把环境相关配置（语言、文件系统、目录覆盖）集中管理，MCP server 和 CLI 只需构造一次并传入所有工具。
- 统一配置更符合 MoonBit 使用结构体的习惯，也便于未来扩展 `snapshot_enabled`、`chip_dir` 等字段。

---

## 将 diff/preview/warnings 作为 CLI 专属功能，不进入 `be_*` 核心逻辑

**提案内容**：`be_replace`/`be_insert`/`be_delete` 只返回最简结果，diff、preview、内容警告由 CLI 层自行计算。

**否决原因**：

- MCP server 同样需要 diff 和 warnings，放在 CLI 层会导致 server 和 CLI 重复实现。
- 内容警告（tab、行尾空白、缺少换行）依赖于写入后的完整内容，在核心逻辑里计算最自然。
- preview 模式涉及“模拟编辑但不写入”，需要和真实编辑共享同一套行处理逻辑，放在核心层能避免行为分叉。
- 统一返回 `EditResult` 结构，CLI 和 server 都能复用。

---

## CLI 子命令只在 native 上完整可用，其他 target 通过条件编译降级/禁用

- AI 报告：main.mbt 使用 @sys.get_cli_args() 和 @sys.exit(code)，它们来自 moonbitlang/x/sys。这个包目前只支持 native（需要真正的 OS 进程参数和退出码）。
- 但是 MoonBit 是多 target 支持的语言；而 moonbitlang/x/sys 只支持 native，因此，我需要让其他平台也能实现核心功能
- 由此考虑，废除在其他平台的 CLI 功能，使用条件编译等类似的思路，在其他 target 上仅提供启动即 MCP 服务器的能力，native target 提供完全能力
- 这个决议不影响各种工具的实现库
